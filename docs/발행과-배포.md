# 발행과 배포 방법 (런북)

이 파일 하나만 보고 그대로 따라 하면 된다. 마지막 업데이트 2026-07-07.

## TL;DR

```bash
cd ~/Projects/skaug12.github.io

# 경우 ①  새 글을 올린다 (옵시디언에 쓴 글을 라이브로)
python3 scripts/publish.py          # 미리보기(뭐가 나갈지만 출력)
python3 scripts/publish.py --go     # 실제 발행 + 배포 + 라이브 검증

# 경우 ②  글은 그대로, 디자인·템플릿만 바꿔서 배포한다
git add -A && git commit -m "설명" && git push origin main   # 소스(main) 저장
python3 scripts/publish.py --deploy                          # 빌드 + gh-pages 배포
```

라이브: https://skaug12.github.io/ · 반영까지 보통 30초~3분.

---

## 자동 발행 (켜져 있음) ⭐

**이제 손으로 명령을 칠 필요 없다.** 옵시디언에 글을 쓰고 frontmatter를 `status: publish`로만 바꾸면, **10분 안에 자동으로** 변환·커밋·빌드·gh-pages 배포까지 된다.

- 원리: launchd 작업 `com.seulki.blog-autopublish`가 10분마다 `publish.py --go`를 실행. 발행할 글이 없으면 조용히 넘어간다(로컬 no-op, 네트워크·git 안 건드림). `status: publish` 글이 있으면 발행하고, 성공하면 옵시디언 원본을 `status: published`로 바꿔서 재발행을 막는다.
- 로그: `logs/autopublish.log` (`tail -f`로 실시간 확인 가능)
- 관리:
  ```bash
  python3 scripts/install_autopublish.py --status      # 등록 상태 확인
  python3 scripts/install_autopublish.py --uninstall   # 자동 발행 끄기
  python3 scripts/install_autopublish.py               # 다시 켜기 (멱등)
  python3 scripts/install_autopublish.py --interval 300  # 주기 5분으로
  ```
- **즉시 올리고 싶으면** 10분 기다리지 말고 직접 `python3 scripts/publish.py --go`.

즉, 평소 흐름은 이것뿐이다: **옵시디언에 쓰기 → `status: publish` → 끝.** 아래 수동 방법은 참고용/문제 해결용.

## 구조 (왜 이렇게 하나)

- **main 브랜치** = Hugo 소스(글·템플릿·CSS). 여기에 커밋한다.
- **gh-pages 브랜치** = `hugo`가 만든 `public/` 결과물. GitHub Pages가 **이 브랜치를 서빙**한다.
- gh 토큰에 `workflow` 스코프가 없어 GitHub Actions 자동배포를 못 쓴다. 그래서 **로컬에서 빌드해 gh-pages로 강제 푸시**한다. (`scripts/publish.py`의 `deploy()`가 이 일을 한다.)
- 나중에 토큰에 workflow 스코프를 주면 `scripts/hugo-workflow.yml.txt`를 `.github/workflows/hugo.yml`로 옮겨 CI 배포로 전환 가능.

## 경우 ① 새 글 발행 (옵시디언 → 라이브)

### 1. 옵시디언에 글 쓰기
`Memo/콘텐츠파이프라인/블로그/` 폴더에 새 `.md` 파일을 만든다. frontmatter:

```yaml
---
title: 글 제목
slug: english-slug          # URL이 된다 → /posts/english-slug/
date: 2026-07-08
category: 인사이트 노트        # 셋 중 하나: 인사이트 노트 / 생각 / 개인 자동화  (필수급)
series: 자동화가 터진 날들      # (선택) 연재명
summary: 한 줄 요약           # 대표글 설명·목록·AEO에 쓰인다
episode: "01"                # (선택) 노트 번호 배지
pinned: true                 # (선택) 홈 대표글로 고정 — 딱 하나만
status: draft
---

본문...

{{< summary >}}
- AEO용 세 줄 정리 (선택)
{{< /summary >}}
```

- `category`를 안 넣으면 목록에서 "글"로만 뜨고 카테고리 카드에 안 잡힌다. **꼭 넣는다.**
- `pinned: true`는 홈 상단 "대표글"이 된다. 하나만 지정.

### 2. 발행 상태로 바꾸기
다 쓰면 frontmatter `status: draft` → `status: publish`.

### 3. 발행
**자동 발행이 켜져 있으면 여기서 끝.** 10분 안에 알아서 나간다. 즉시 올리고 싶을 때만 아래를 직접 실행:
```bash
cd ~/Projects/skaug12.github.io
python3 scripts/publish.py          # dry-run: 발행 대상 목록만 확인
python3 scripts/publish.py --go     # 실제 발행
```

`--go`가 자동으로 하는 일: 옵시디언 초안 → `content/posts/{slug}.md` 변환 → main 커밋·푸시 → `hugo --minify` 빌드 → gh-pages 강제 푸시 → 라이브 URL 검증(최대 5분) → 성공 시 **옵시디언 원본** frontmatter에 `status: published` + `posted_url` 기록. **원본 본문은 절대 안 건드린다(frontmatter만).**

## 경우 ② 디자인·템플릿만 배포 (새 글 없이)

CSS/레이아웃/설정만 고쳤을 때.

```bash
cd ~/Projects/skaug12.github.io
git add -A
git commit -m "무엇을 바꿨는지"
git push origin main            # 소스 저장 (main)
python3 scripts/publish.py --deploy   # 빌드 + gh-pages 배포
```

`--deploy`는 `hugo --minify` 빌드 후 `public/`을 gh-pages로 강제 푸시만 한다(글 발행 로직 없음).

## 문제 해결

- **배포했는데 라이브가 안 바뀜**: Pages 재빌드에 30초~3분. 잠시 후 캐시 우회 curl로 재확인. 그래도 안 되면 GitHub 레포 → Settings → Pages → 빌드 상태 확인.
- **폰트가 404**: gh-pages에 아직 반영 전이거나, `static/fonts/PretendardVariable.woff2`가 커밋 안 됨. `git status`로 확인.
- **`publish.py --go`가 "발행 대상 없음"**: 옵시디언 글의 `status`가 `publish`인지, `slug`가 있는지 확인.
- **push 인증 실패**: gh 로그인/자격증명 만료. `gh auth status` 확인.
- **수동 배포(스크립트가 안 될 때)**:
  ```bash
  hugo --minify
  cd public && touch .nojekyll
  git init -b gh-pages -q && git add -A && git commit -q -m deploy
  git push -f https://github.com/skaug12/skaug12.github.io.git gh-pages
  rm -rf .git && cd ..
  ```

## 나중에 도메인 붙일 때

1. `static/CNAME`에 도메인 한 줄 추가
2. `hugo.toml`의 `baseURL` + `scripts/publish.py`의 `BASE_URL` 둘 다 수정
3. DNS: A 레코드 4개(185.199.108~111.153) + www CNAME → skaug12.github.io
4. 레포 Settings → Pages → Custom domain 설정 + Enforce HTTPS
